## Ключевые выводы

### Самый важный результат
**На больших массивах (100k) с JIT-прогревом Holey массивы значительно быстрее Regular для `push` — в 2.4 раза (0.93 против 2.25).** Без JIT они равны. Это подтверждает предыдущий паттерн.

### Другие важные наблюдения

1. **`push` на 100 без JIT — аномалия:**
   - Holey показывает 6.98 мс против 1.31 у Regular
   - JIT исправляет это, ускоряя Holey в **4.3 раза**
   - Вероятно, начальная оптимизация Holey требует прогрева

2. **`pop` — Regular стабильно быстрее на больших размерах:**
   - Преимущество ~46% на 100k независимо от JIT
   - На малых размерах разница минимальна

3. **`shift` — Holey неожиданно хорош:**
   - На 1000 без JIT Holey быстрее на 16%
   - С JIT преимущество сохраняется (~11%)
   - На 10000 — полное равенство

4. **`unshift` — JIT даёт ускорение:**
   - На 100000 ускорение 8-10%
   - Holey и Regular практически неразличимы
   - Операция остаётся крайне медленной (2300-2500 мс)

5. **Критическое наблюдение по Regular push на 100k:**
   - Без JIT: 0.93 мс (быстро)
   - **С JIT: 2.25 мс (медленно в 2.4 раза)**
   - JIT **ухудшил** производительность Regular для этой операции!


### Главные выводы

1. **JIT-прогрев критичен для Holey массивов** — без него `push` на 100 может быть в 5 раз медленнее

2. **Regular push на 100k деградирует после JIT** — с 0.93 до 2.25 мс (редкий случай ухудшения)

3. **Holey массивы не всегда медленнее** — для `push` на больших объёмах и `shift` на средних они показывают лучшие результаты

4. **`shift` и `unshift` остаются узким местом** — для частых операций вставки/удаления в начало массива нужны другие структуры данных (связные списки, deque)

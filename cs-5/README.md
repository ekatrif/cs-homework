# Анализ производительности различных реализаций PixelStream

| 3 | ArrayOfObjects | 0/8 | ⭐⭐ |

## Ключевые моменты

### 1. RowMajor vs ColMajor

Все реализации быстрее при обходе по строкам (RowMajor):

| Реализация | RowMajor | ColMajor | Разница |
|------------|----------|----------|---------|
| FlatArray | 2.36ms | 2.69ms | +14% |
| ArrayOfArrays | 5.94ms | 14.73ms | +148% |

**Почему:** процессоры оптимизированы для последовательного доступа к памяти (CPU cache).

### 2. FlatArray vs TypedArray

Ожидалось, что TypedArray будет быстрее везде, но результаты оказались неожиданными:

| Операция | FlatArray | TypedArray |
|----------|-----------|------------|
| setPixel | 2.75ms ✅ | 7.63ms |
| forEach | 2.36ms | 1.79ms ✅ |

**Объяснение:**
- `forEach` в TypedArray хорошо оптимизирован JIT-компилятором
- `setPixel` страдает от проверки границ (0-255) и преобразования типов
**Почему forEach быстрее в TypedArray:**
- JIT-компилятор лучше оптимизирует итерации по TypedArray
- Предсказуемая структура памяти позволяет применять агрессивные оптимизации
- Отсутствие необходимости в проверке типа элементов (всегда числа)
**Почему setPixel медленнее в TypedArray:**
- Каждое присвоение `this.data[idx] = rgba[0]` приводит число к 8-битному целому
- Выполняется проверка диапазона (0-255)
- TypedArray имеет фиксированную длину, проверка границ происходит при каждом доступе

### 3. Стоимость создания объектов

```javascript
// ArrayOfObjects - каждый setPixel создаёт новый объект
setPixel() {
  this.data[y][x] = { r, g, b, a };
}
// Результат: 10.88ms

// FlatArray - просто меняет число в массиве
setPixel() {
  this.data[idx] = r;
  this.data[idx + 1] = g;
}
// Результат: 2.75ms
```

## AI-рекомендации по выбору

| Сценарий | Лучший выбор | Причина |
|----------|--------------|---------|
| Много записей (setPixel) | `FlatArray` | В 2.8x быстрее TypedArray |
| Много чтений (getPixel) | `FlatArray` | Почти одинаков с ArrayOfObjects |
| Итерации без модификаций | `TypedArray` | На 32% быстрее FlatArray |
| Смешанные операции | `FlatArray` | Лучший баланс |
| Работа с WebGL/Canvas | `TypedArray` | Совместимость с буферами |
| Читаемость кода | `ArrayOfObjects` | Понятный код, но медленный |
| Никогда не используйте | `ArrayOfArrays` | Медленный везде |

# Анализ производительности реализаций работы с пикселями (разные варианты разрешения)

## Общие выводы

**Размер данных критически влияет на лидера производительности:**
- Маленькие изображения (500×500) → FlatArray побеждает в 5 из 8 тестов
- Средние изображения (1000×1000) → TypedArray побеждает в 4 из 8 тестов
- Крупные изображения (5000×5000) → FlatArray побеждает в 5 из 8 тестов

**Безусловный аутсайдер:** ArrayOfArrays показывает наихудшие результаты во всех тестах и при всех размерах.

## Ключевые наблюдения

### FlatArray
- **Сильные стороны:** запись, комбинированные операции, стабильность
- **Слабые стороны:** практически отсутствуют
- **Вердикт:** лучший выбор для большинства сценариев

### TypedArray
- **Сильные стороны:** итерации (особенно column-major), чтение
- **Слабые стороны:** комбинированные операции, запись на больших данных
- **Вердикт:** оптимален для аналитики и обработки без модификаций

### ArrayOfObjects
- **Сильные стороны:** чтение на средних и больших данных
- **Слабые стороны:** запись (в 10 раз медленнее FlatArray на 5000×5000)
- **Вердикт:** только для read-heavy сценариев

### ArrayOfArrays
- **Сильные стороны:** отсутствуют
- **Слабые стороны:** всё
- **Вердикт:** никогда не использовать

---

## Рекомендации AI

### Используйте FlatArray если:
- Приоритетны операции записи или чтения+записи
- Нужна стабильная производительность при любом размере
- Работаете с большими изображениями (5000×5000+)

### Используйте TypedArray если:
- Приоритетны итерации без модификации
- Работаете со средними изображениями (1000×1000)
- Важна производительность column-major доступа

### Используйте ArrayOfObjects только если:
- Исключительно чтение пикселей
- Запись практически отсутствует

### Никогда не используйте ArrayOfArrays:
- Худший во всех тестах
- На больших данных до 38 раз медленнее лидера
- Column-major доступ делает его полностью непригодным
